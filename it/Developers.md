Se un set di dati fallisce durante l&#39;integrazione, viene creato un problema su Taiga (piattaforma di gestione del progetto). Il problema mostra l&#39;errore sollevato e lo sviluppatore dovrebbe risolverlo e spingere nuovamente il suo lavoro su Git in modo che lo script possa provare a integrarlo di nuovo la sera successiva. </p><p> Il codice sorgente dello script di integrazione è disponibile a questo link: <a href="https://github.com/HotMaps/CI_DatasetIntegration">Integrazione dati</a> </p><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h2> <a class="anchor" id="add-your-datasets-on-gitlab" href="#add-your-datasets-on-gitlab"><i class="fa fa-link"></i></a> Aggiungi i tuoi set di dati su GitLab </h2><p> Per aggiungere set di dati all&#39;elenco dei set di dati pubblici, devono essere inviati a un nuovo repository Git su GitLab. Ecco l&#39;organizzazione GitLab in cui devono essere <a href="https://gitlab.com/hotmaps">inviati i set di dati</a> : <a href="https://gitlab.com/hotmaps">Set di dati su GitLab</a> . </p><p> Una volta al giorno i repository vengono controllati per nuovi commit e integrati in tal caso. Il processo di integrazione verifica se i dati sono conformi o meno alla specifica. </p><p> Ecco le specifiche: <a href="https://wiki.hotmaps.eu/uploads/Hotmaps_Data-upload-on-Gitlab_2017-12-04_V4.pdf">Hotmaps_Data-upload-on-Gitlab_2017-12-04_V4.pdf</a> </p><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h2> <a class="anchor" id="list-of-main-repositories" href="#list-of-main-repositories"><i class="fa fa-link"></i></a> Elenco dei principali repository </h2><p> Il codice dell&#39;applicazione si trova su GitHub sotto l&#39; <a href="https://github.com/HotMaps">organizzazione Hotmaps</a> . Questa organizzazione possiede diversi repository </p><ul><li> <a href="https://github.com/HotMaps/Hotmaps-toolbox-service">Hotmaps-toolbox-client</a> contiene il frontend della nostra applicazione. È un progetto angolare (JavaScript) </li><li> <a href="https://github.com/HotMaps/Hotmaps-toolbox-service">Hotmaps-toolbox-service</a> contiene l&#39;API per la nostra applicazione. È basato su Flask (Python) </li><li> <a href="https://github.com/HotMaps/base_calculation_module">modulo di calcolo di</a> base è il modello di base che puoi utilizzare per creare i tuoi moduli di calcolo per Hotmap </li><li> un elenco di moduli di calcolo </li></ul><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h2> <a class="anchor" id="how-to-contribute-code" href="#how-to-contribute-code"><i class="fa fa-link"></i></a> Come contribuire al codice </h2><p> Se vuoi aggiungere del codice a Hotmaps hai due possibilità: se vuoi aggiornare direttamente l&#39;interfaccia o il backend, devi modificare il client o il repository del servizio del toolbox. Se vuoi aggiungere il tuo modulo di calcolo, puoi creare il tuo repository seguendo <a href="https://github.com/HotMaps/base_calculation_module">il readme del repository base_calculation_module</a> </p><p> Se vuoi eseguire del lavoro sul repository Git, non lavorare direttamente con il ramo master. Crea un nuovo ramo dal ramo di sviluppo, fai il tuo lavoro su questo ramo e quando la tua funzionalità viene testata, puoi unire il tuo lavoro con il ramo di sviluppo come mostrato nel grafico seguente. </p><p> <strong>Git_workflow</strong> </p><p> Per inviare qualcosa ad alcuni repository Hotmaps devi essere un membro del team Hotmaps, se non lo sei puoi ancora eseguire un fork del nostro strumento per sviluppare il tuo strumento. </p><p> Puoi trovare ulteriori informazioni su come lavorare in questi documenti: </p><ul><li> <a href="https://wiki.hotmaps.eu/uploads/Hotmaps_python_best_practices_tutorial_2017-08-07_v01.pdf">Hotmaps_python_best_practices_tutorial_2017-08-07_v01.pdf</a> </li><li> <a href="https://wiki.hotmaps.eu/uploads/Hotmaps_Testing_in_python_tutorial_pytest_2017-08-07_v01.pdf">Hotmaps_Testing_in_python_tutorial_pytest_2017-08-07_v01.pdf</a> </li><li> <a href="https://wiki.hotmaps.eu/uploads/GitFlow_Guidelines_CREM_2017-02-01.pdf">GitFlow_Guidelines_CREM_2017-02-01.pdf</a> </li></ul><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h2> <a class="anchor" id="description-of-it-infrastructure" href="#description-of-it-infrastructure"><i class="fa fa-link"></i></a> Descrizione dell&#39;infrastruttura IT </h2><p> <strong>ReverseProxy_architecture_latest</strong> </p><p> Tutti i servizi e i componenti vengono utilizzati tramite il proprio contenitore Docker. Tutti questi contenitori sono definiti in un unico file docker-compose. L&#39;immagine sopra rappresenta l&#39;architettura IT di Hotmaps. </p><p> Alcune organizzazioni partner limitano la comunicazione solo alla porta 80. Per evitare i problemi causati da questa limitazione, è stata realizzata la creazione di un proxy inverso. Questo proxy inverso offre un unico punto di ingresso e quindi distribuisce la richiesta inviata dal cliente al servizio interessato. Il proxy inverso è composto da tre componenti: </p><ol><li> Server proxy inverso: funge da punto di ingresso univoco e distribuisce le richieste ai servizi giusti. </li><li> Proxy-gen: è un servizio che mappa automaticamente tutti i servizi nel proxy inverso. Pertanto, non è necessario aggiungere manualmente un nuovo servizio alla configurazione del proxy </li><li> lets-encrypt: è un servizio che consente l&#39;utilizzo del protocollo SSL. È necessario per attivare il protocollo https. I certificati SSL sono firmati da un indirizzo e-mail configurato in questo servizio. </li></ol><p> Esistono tre reti: </p><ul><li> hotmaps_nginx consente al proxy inverso di comunicare con l&#39;API, il frontend e il GeoServer. Consente principalmente di distribuire una richiesta al servizio corretto tra i tre. </li><li> hotmaps_backend permette la comunicazione tra tutti i componenti del backend: API, frontend, GeoServer e il database PostgreSQL. </li><li> hotmaps_cm-net consente la comunicazione tra ogni modulo di calcolo e l&#39;API. </li></ul><p> Ogni modulo di calcolo ha il proprio contenitore Docker. </p><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h3> <a class="anchor" id="run-with-docker" href="#run-with-docker"><i class="fa fa-link"></i></a> Esegui con Docker </h3><p> HotMaps utilizza <a href="https://www.docker.com/">Docker</a> software e <a href="https://docs.docker.com/compose/">Docker-Compose</a> strumento per gestire i contenitori. Un file docker-compose.yml contiene l&#39;intera configurazione dell&#39;architettura Docker (configurazione di contenitori, reti, collegamenti, ...). Ciò consente ai contenitori di essere eseguiti con un semplice comando: </p><pre> <code class="language-shell">docker-compose up</code> </pre><p> <em>Altre informazioni su docker-compose sono disponibili sul sito Web di Docker: <a href="https://docs.docker.com/compose/reference/">riferimento alla riga di comando di</a> <a href="https://docs.docker.com/compose/compose-file/">Compose</a> e <a href="https://docs.docker.com/compose/compose-file/">riferimento al file di composizione</a> .</em> </p><p> C&#39;è un solo contenitore che viene eseguito separatamente dagli altri: è il database perché deve rimanere sempre attivo. Ecco perché non si trova nel file di configurazione docker-compose. </p><p> <a href="#table-of-contents"><strong><code>To Top</code></strong></a> </p> <h3> <a class="anchor" id="server-infrastructure" href="#server-infrastructure"><i class="fa fa-link"></i></a> Infrastruttura server </h3><h4> <a class="anchor" id="infrastructure" href="#infrastructure"><i class="fa fa-link"></i></a> Infrastruttura </h4><p> Per il momento, il server è ospitato presso l&#39;HES-SO in Svizzera.
